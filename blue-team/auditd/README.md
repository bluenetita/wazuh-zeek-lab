# Auditd

This directory contains the Linux Audit configuration deployed on `ClientVM`, the Linux endpoint used as the primary victim machine in the laboratory.

Auditd provides host-based visibility into security-relevant operating-system activity and complements the network-based telemetry generated by Zeek.

## Directory structure

```text
auditd/
├── README.md
└── rules.d/
    ├── audit.rules
    ├── privileged-command.rules
    └── reverse_shell.rules
```

These files correspond to the persistent Auditd rules stored on `ClientVM` under:

```text
/etc/audit/rules.d/
```

## Role in the laboratory

Auditd is used to collect information about activities performed directly on `ClientVM`, including:

* successful outbound connection attempts;
* execution of privileged commands;
* use of `sudo` and `su`;
* execution of SUID programs;
* commands executed from root login sessions;
* execution of programs from the monitored download directory.

Audit events are written to:

```text
/var/log/audit/audit.log
```

The Wazuh Agent installed on `ClientVM` collects the Auditd log and forwards the events to the Wazuh Manager for analysis and alert generation.

## Event flow

```text
Process on ClientVM
        |
        v
Linux Audit subsystem
        |
        v
auditd
        |
        v
/var/log/audit/audit.log
        |
        v
Wazuh Agent
        |
        v
Wazuh Manager
```

For the reverse-shell scenario, host-based Auditd events are compared with the network-based events generated by Zeek:

```text
ClientVM / Auditd                  ZeekVM
        |                             |
        v                             v
  rs_connect event         Reverse-shell network event
        |                             |
        +--------------+--------------+
                       |
                       v
                 Wazuh Manager
```

## Rule files

### `audit.rules`

This file contains the base configuration of the Linux Audit subsystem.

It performs the following operations:

* deletes previously loaded Audit rules;
* configures the kernel Audit event buffer;
* sets the maximum backlog waiting time;
* configures the Audit failure mode.

The following directive removes all previously loaded rules:

```text
-D
```

For this reason, `audit.rules` must be processed before the monitoring rules contained in the other files.

The current base configuration uses:

```text
-b 8192
--backlog_wait_time 60000
-f 1
```

Where:

* `-b 8192` configures the maximum number of outstanding Audit buffers;
* `--backlog_wait_time 60000` defines how long the kernel can wait when the Audit backlog is full;
* `-f 1` reports Audit failures through kernel logging without stopping the system.

### `privileged-command.rules`

This file monitors privileged command execution and other security-relevant process activity.

The following Audit keys are used:

| Audit key       | Purpose                                                                  |
| --------------- | ------------------------------------------------------------------------ |
| `sudo_exec`     | Execution of `/usr/bin/sudo` by an authenticated user                    |
| `su_exec`       | Execution of `/usr/bin/su` by an authenticated user                      |
| `suid_exec`     | Execution of a program with effective root privileges by a non-root user |
| `root_exec`     | Command execution from a direct root login session                       |
| `download_exec` | Execution of programs located under `/home/client/Downloads`             |

Both 64-bit and 32-bit system-call architectures are monitored through:

```text
arch=b64
arch=b32
```

These events are particularly useful during the privilege-escalation and post-exploitation phases.

### `reverse_shell.rules`

This file contains the Audit rules used to increase host-based visibility during the reverse-shell scenario.

The active rules monitor successful `connect()` system calls generated by processes associated with an authenticated user session.

The events are tagged with:

```text
rs_connect
```

The active rules are:

```text
-a always,exit -F arch=b64 -S connect -F success=1 -F auid>=1000 -F auid!=4294967295 -k rs_connect
-a always,exit -F arch=b32 -S connect -F success=1 -F auid>=1000 -F auid!=4294967295 -k rs_connect
```

The filters have the following purpose:

| Filter             | Meaning                                                           |
| ------------------ | ----------------------------------------------------------------- |
| `always,exit`      | Evaluate and record the system call when it terminates            |
| `arch=b64`         | Monitor 64-bit system calls                                       |
| `arch=b32`         | Monitor 32-bit system calls                                       |
| `-S connect`       | Monitor attempts to connect a socket to a remote or local address |
| `success=1`        | Record only successful connections                                |
| `auid>=1000`       | Include sessions originally associated with regular users         |
| `auid!=4294967295` | Exclude events whose Audit user ID is unset                       |
| `-k rs_connect`    | Associate the event with the `rs_connect` Audit key               |

Depending on the generated Audit records, the event can provide information such as:

* executable path;
* process name;
* process identifier;
* parent process identifier;
* user identifier;
* effective user identifier;
* Audit user identifier;
* system-call result;
* socket address;
* destination address and port.

The file also contains commented rules that were evaluated during laboratory development for monitoring:

* process execution through `execve`;
* shell execution;
* interpreter execution;
* common tools such as Netcat, Socat and Telnet.

Commented rules are retained for documentation purposes but are not active in the current configuration.

## Deployment

Copy the rule files to the persistent Auditd rules directory:

```bash
sudo cp rules.d/*.rules /etc/audit/rules.d/
```

Check whether the persistent rules have changed:

```bash
sudo augenrules --check
```

Load the rules:

```bash
sudo augenrules --load
```

Verify the active rules:

```bash
sudo auditctl -l
```

Verify the Audit subsystem status:

```bash
sudo auditctl -s
```

Check the Auditd service:

```bash
sudo systemctl status auditd
```

## Event verification

Search for reverse-shell connection events:

```bash
sudo ausearch -k rs_connect
```

Display interpreted values:

```bash
sudo ausearch -k rs_connect -i
```

Search for privileged-command events:

```bash
sudo ausearch -k sudo_exec -i
sudo ausearch -k su_exec -i
sudo ausearch -k suid_exec -i
sudo ausearch -k root_exec -i
sudo ausearch -k download_exec -i
```

Monitor the Audit log in real time:

```bash
sudo tail -f /var/log/audit/audit.log
```

## Detection scope

The `rs_connect` rules provide telemetry about successful connection system calls, but a successful `connect()` call does not by itself demonstrate the presence of a reverse shell.

Legitimate applications such as browsers, package managers, monitoring agents and command-line network tools also generate `connect()` events.

The event must therefore be evaluated together with:

* the executable responsible for the connection;
* its parent process;
* the destination address and port;
* the event timestamp;
* the user session;
* the corresponding Zeek connection;
* other alerts generated by Wazuh.

Auditd provides process-level context, while Zeek provides network-level context. Combining these two perspectives improves the reliability of the analysis.

## Current limitations

The reverse-shell rules only monitor successful connections because they use:

```text
-F success=1
```

Failed connection attempts are therefore excluded.

The rules also require:

```text
-F auid>=1000
-F auid!=4294967295
```

This reduces events generated by system services but can exclude reverse shells started by:

* compromised daemons;
* system services;
* processes without an authenticated login session;
* processes whose Audit user ID is unset.

The Auditd event does not contain the Zeek connection UID. Correlation between host-based and network-based events must therefore rely on shared attributes such as:

* source host;
* destination address;
* destination port;
* timestamp;
* process information;
* analysis time window.

## Security and sanitization

The repository contains configuration files only.

The following information must not be committed:

* complete unredacted Audit logs;
* credentials;
* private keys;
* authentication tokens;
* sensitive usernames;
* personal information;
* public IP addresses;
* malicious payloads;
* confidential file contents.
