# Wazuh Custom Rules

This directory contains the custom Wazuh rules deployed on the Wazuh Manager for the laboratory.

The rules analyze telemetry generated by:

* Zeek standard logs;
* custom Zeek scripts;
* Linux Auditd;
* Wazuh File Integrity Monitoring;
* multi-source correlations between network and endpoint events.

The main objective is to detect and reconstruct security scenarios such as:

* suspicious DNS and network activity;
* rejected connection bursts;
* reverse-shell execution;
* malware download and execution;
* privilege escalation;
* execution from monitored directories;
* possible data exfiltration;
* complete attack lifecycles observed from both the endpoint and the network.

## Directory structure

```text
rules/
├── README.md
├── 001_zeek_rules.xml
├── 002_zeek_rules_custom.xml
├── 003_auditd_rev_shell.xml
├── 004_zeek_auditd_correlation.xml
└── 1004_auditd_rules.xml
```

Only active custom rule files are included in this directory.

Backup files, disabled rules and temporary test configurations are not part of the documented ruleset.

---

## Detection architecture

The rules combine two complementary visibility sources.

### Network-based visibility

Zeek analyzes mirrored network traffic and produces:

* standard JSON logs;
* custom reverse-shell events;
* possible malware-download events;
* data-exfiltration events.

### Host-based visibility

Auditd and the Wazuh Agent provide information about:

* executed processes;
* process identifiers;
* parent processes;
* authenticated users;
* effective privileges;
* outbound `connect()` system calls;
* execution of SUID programs;
* use of `sudo` and `su`;
* commands executed from root sessions;
* execution from monitored directories.

### Combined processing flow

```text
                         Network traffic
                               |
                               v
                             Zeek
                               |
                    Standard and custom logs
                               |
                               v
+------------------------ Wazuh Manager ------------------------+
|                                                               |
|  ClientVM                                                     |
|     |                                                         |
|     +--> Auditd --> /var/log/audit/audit.log --> Wazuh Agent |
|     |                                                         |
|     +--> Wazuh FIM ----------------------------> Wazuh Agent  |
|                                                               |
+-----------------------------+---------------------------------+
                              |
                              v
                     Decoders and base rules
                              |
                              v
                    Multi-source correlations
                              |
                              v
                         Wazuh alerts
```

# Rule files

## `001_zeek_rules.xml`

This file contains the base rules used to analyze standard Zeek JSON events.

The rules belong to the following group:

```text
zeek
```

### Rule summary

| Rule ID  | Level | Purpose                                                       |
| -------- | ----: | ------------------------------------------------------------- |
| `100900` |     0 | Base rule for JSON events                                     |
| `100901` |     5 | Detects DNS queries                                           |
| `100902` |     0 | Suppresses common mDNS traffic                                |
| `100903` |     7 | Detects rejected connections                                  |
| `100904` |    10 | Detects multiple rejected connections in a short interval     |
| `100905` |     0 | Suppresses DNS queries to the Wazuh CTI domain                |
| `100906` |     8 | Detects connections to servers using self-signed certificates |
| `100907` |    12 | Detects connections to servers using expired certificates     |

### Base JSON rule

Rule `100900` is the parent rule used by the other Zeek JSON detections:

```xml
<rule id="100900" level="0">
  <decoded_as>json</decoded_as>
  <description>Json alerts</description>
</rule>
```

The rule does not generate a visible alert because it uses level `0`. Its purpose is to identify JSON events and provide a common parent for more specific rules.

### DNS query detection

Rule `100901` detects DNS queries directed to destination port `53`.

The alert includes information such as:

* DNS query;
* source address;
* source port;
* resolved IP address.

### mDNS suppression

Rule `100902` identifies traffic originating from source port `5353`.

This traffic is normally associated with multicast DNS and is assigned level `0` to reduce unnecessary alerts.

### Rejected connections

Rule `100903` detects Zeek connections whose state is:

```text
REJ
```

Rule `100904` extends this detection and triggers when at least five rejected connections are observed within twenty seconds.

This pattern may indicate:

* port scanning;
* service discovery;
* repeated connection attempts;
* misconfigured applications.

### Wazuh CTI exclusion

Rule `100905` suppresses DNS queries to:

```text
cti.wazuh.com
```

This prevents normal Wazuh threat-intelligence activity from generating unnecessary DNS alerts.

### TLS certificate detections

Rules `100906` and `100907` identify:

* self-signed certificates;
* expired certificates.

These conditions are not always malicious, but they may indicate:

* improperly configured services;
* untrusted internal services;
* interception infrastructure;
* obsolete or abandoned systems.

## `002_zeek_rules_custom.xml`

This file contains rules for the logs generated by the custom Zeek scripts developed for the laboratory.

The rules cover:

* possible malware downloads;
* reverse-shell activity;
* possible data exfiltration.

## Reverse-shell rules

### Rule summary

| Rule ID  | Level | Event type               | Purpose                                                          |
| -------- | ----: | ------------------------ | ---------------------------------------------------------------- |
| `100909` |     5 | `possible_malware`       | Detects a file classified by the Zeek script as possible malware |
| `100910` |     5 | `reverse_shell_live`     | Detects the possible beginning of a reverse-shell connection     |
| `100911` |     5 | `reverse_shell_final`    | Detects the closure of the possible reverse-shell connection     |
| `100912` |     5 | `reverse_shell_movement` | Detects traffic observed during the possible reverse shell       |

### Monitored log files

```text
/var/log/zeek-custom/possible_malware.log
/var/log/zeek-custom/reverse_shell_live.log
/var/log/zeek-custom/reverse_shell_final.log
/var/log/zeek-custom/reverse_shell_movement.log
```

### Possible malware event

Rule `100909` processes events containing:

```text
event_type = possible_malware
```

The alert can include:

* filename;
* source address;
* destination address;
* destination port;
* MIME type;
* transferred bytes;
* SHA-1 hash;
* SHA-256 hash;
* additional notes generated by the Zeek script.

The event is assigned to groups including:

```text
zeek_custom
malware
reverse_shell_stage
```

### Reverse-shell start

Rule `100910` processes:

```text
event_type = reverse_shell_live
```

The alert includes:

* Zeek connection UID;
* source address;
* destination address;
* destination port;
* Zeek weird or payload indicator.

This event represents the initial network-based evidence of a possible reverse shell.

### Reverse-shell movement

Rule `100912` processes:

```text
event_type = reverse_shell_movement
```

The alert includes:

* connection UID;
* connection duration;
* origin bytes;
* response bytes;
* origin packets;
* response packets.

This event provides evidence that the connection remained active and exchanged traffic.

### Reverse-shell closure

Rule `100911` processes:

```text
event_type = reverse_shell_final
```

It is generated when Zeek observes the end of the monitored connection.

The event contains the final connection statistics and can be used to complete the reverse-shell lifecycle.

## Data-exfiltration rules

### Rule summary

| Rule ID  | Level | Event type                   | Purpose                                              |
| -------- | ----: | ---------------------------- | ---------------------------------------------------- |
| `100913` |     3 | `baseline_completed`         | Records completion of the learning phase             |
| `100914` |     8 | `possible_data_exfiltration` | Detects outgoing traffic above the dynamic threshold |

Both rules process:

```text
/var/log/zeek-custom/data_exfiltration.log
```

### Baseline completion

Rule `100913` records when the Zeek script completes its baseline and calculates the dynamic threshold for the monitored source host.

This is an informational event and does not independently indicate malicious activity.

### Possible data exfiltration

Rule `100914` triggers when outgoing traffic exceeds the threshold calculated by the Zeek script.

The event includes:

* source address;
* destination address;
* destination port.

This detection must be evaluated together with:

* the monitored host role;
* the destination;
* the amount of data transferred;
* normal host behavior;
* Auditd file-access events;
* Wazuh FIM events;
* other endpoint evidence.

## `003_auditd_rev_shell.xml`

This file contains the host-based rule used to analyze successful outbound network connections recorded by Auditd.

The rule belongs to the following groups:

```text
auditd
reverse_shell
reverse_shell_endpoint
connect
```

### Rule summary

| Rule ID  | Level | Purpose                                                         |
| -------- | ----: | --------------------------------------------------------------- |
| `110900` |     5 | Detects a successful outbound `connect()` syscall from ClientVM |

### Decoder dependency

The rule requires the following decoder:

```text
auditd_rs_connect_enriched
```

The decoder is stored in:

```text
blue-team/wazuh/decoders/000_audit_saddr_decoder.xml
```

The corresponding endpoint Auditd rules are stored in:

```text
blue-team/auditd/rules.d/reverse_shell.rules
```

### Conditions

Rule `110900` verifies:

```text
success = yes
syscall = 42
destination port != 0
```

It also excludes the following destination addresses:

```text
127.0.0.1
127.0.0.53
10.3.20.2
```

These exclusions reduce events generated by:

* local loopback traffic;
* the local systemd-resolved DNS stub;
* connections directed back to ClientVM itself.

### Extracted context

The resulting alert can include:

* executable filename;
* complete executable path;
* process identifier;
* parent process identifier;
* user identifiers;
* terminal;
* Audit session;
* destination address;
* destination port.

### Detection meaning

A successful `connect()` syscall does not independently prove the existence of a reverse shell.

Legitimate applications also establish outbound connections, including:

* browsers;
* package managers;
* monitoring agents;
* DNS clients;
* update services;
* command-line tools.

The event becomes more significant when correlated with:

* a Zeek `reverse_shell_live` event;
* execution of a recently downloaded file;
* suspicious process ancestry;
* unusual destination ports;
* reverse-shell movement;
* reverse-shell closure.

### Architecture consideration

The rule currently checks:

```text
syscall = 42
```

This corresponds to `connect()` in the x86-64 syscall table used by the laboratory host.

The endpoint Auditd configuration also includes a 32-bit rule. Events produced by a 32-bit process may use a different syscall number and may therefore require an additional Wazuh condition.

## `004_zeek_auditd_correlation.xml`

This file contains the multi-source correlations between:

* Zeek possible-malware events;
* Zeek reverse-shell events;
* Auditd outbound connection events.

The rules use the range:

```text
120900-120926
```

The correlations support different event orders because the endpoint event and the Zeek event can reach the Wazuh Manager in different sequences.

## Correlation mechanisms

The rules use Wazuh correlation directives such as:

```xml
<if_matched_sid>...</if_matched_sid>
<if_sid>...</if_sid>
<same_field>...</same_field>
<different_field>...</different_field>
<global_frequency />
```

### `if_matched_sid`

Checks whether a previous rule has matched within the configured timeframe.

### `if_sid`

Checks the current event against another rule in the correlation chain.

### `same_field`

Requires two events to contain the same value for a field.

Examples include:

```text
dest_ip
dest_port
file_name
uid
```

### `different_field`

Requires two events to contain different values for a field.

This is used to distinguish the malware-download destination from the later reverse-shell destination.

### `global_frequency`

Allows the correlation to operate across events received from different Wazuh agents.

This is required because:

* Zeek events are received from `ZeekVM`;
* Auditd events are received from `ClientVM`.

Without global correlation, Wazuh could treat the two event sources as unrelated.

# Reverse-shell correlation chains

## Chain 1 — Malware downloaded from a different IP

Rules:

```text
120900-120906
```

This chain covers a possible malware file downloaded from one host and later connecting to a different destination.

### Initial correlation

Rule `120900` correlates:

```text
Rule 100909 — possible malware download
Rule 110900 — Auditd outbound connect
```

Requirements include:

* same filename;
* different destination IP;
* events observed within 900 seconds.

### First event order

```text
Possible malware download
        |
        v
Auditd outbound connect
        |
        v
Rule 120900
        |
        v
Zeek reverse_shell_live
        |
        v
Rule 120901
        |
        v
Zeek reverse_shell_movement
        |
        v
Rule 120902
        |
        v
Zeek reverse_shell_final
        |
        v
Rule 120903
```

### Alternative event order

Rules `120904-120906` support the case in which the Zeek reverse-shell start is processed before the malware-and-connect correlation.

### Severity progression

| Stage                                            |                 Rule | Level |
| ------------------------------------------------ | -------------------: | ----: |
| Malware file and endpoint connection correlation |             `120900` |     5 |
| Reverse-shell start                              | `120901` or `120904` |     8 |
| Reverse-shell movement                           | `120902` or `120905` |    10 |
| Complete lifecycle                               | `120903` or `120906` |    12 |

---

## Chain 2 — Reverse shell without malware evidence

Rules:

```text
120907-120912
```

This chain does not require a previous possible-malware download.

It correlates:

```text
Auditd outbound connect
+
Zeek reverse_shell_live
+
Zeek reverse_shell_movement
+
Zeek reverse_shell_final
```

### Supported event orders

The first order is:

```text
Auditd connect
        |
        v
Zeek reverse_shell_live
```

and uses rule `120907`.

The alternative order is:

```text
Zeek reverse_shell_live
        |
        v
Auditd connect
```

and uses rule `120910`.

### Chain progression

| Stage                                       | First order | Alternative order | Level |
| ------------------------------------------- | ----------: | ----------------: | ----: |
| Endpoint connection and reverse-shell start |    `120907` |          `120910` |     8 |
| Network movement                            |    `120908` |          `120911` |    10 |
| Complete lifecycle                          |    `120909` |          `120912` |    12 |

This chain is useful when:

* the payload was already present on the endpoint;
* the malware-download phase was not observed;
* the payload was transferred through an unmonitored channel;
* the reverse shell was created using a legitimate installed tool;
* the connection originated from an interpreter or shell.

---

## Chain 3 — Same destination IP, different filename

Rules:

```text
120913-120919
```

This chain covers a possible malware download and a subsequent endpoint connection involving the same destination IP but a different executable filename.

The initial rule requires:

```text
same dest_ip
different file_name
```

The rule excludes:

```text
/wget
/curl
```

This exclusion reduces the possibility of correlating the original downloader connection with the later execution of the downloaded payload.

### Supported orders

Rules `120913-120916` cover the first event order.

Rules `120917-120919` cover the alternative order in which the reverse-shell start is received before the initial correlation is completed.

### Severity progression

| Stage                                   | Rules                | Level |
| --------------------------------------- | -------------------- | ----: |
| Same-IP malware and endpoint connection | `120913`             |     5 |
| Reverse-shell start                     | `120914` or `120917` |     8 |
| Reverse-shell movement                  | `120915` or `120918` |    10 |
| Complete lifecycle                      | `120916` or `120919` |    12 |

---

## Chain 4 — Same destination IP and same filename

Rules:

```text
120920-120926
```

This chain represents the strongest malware-related correlation.

It requires:

```text
same dest_ip
same file_name
```

The correlation links:

* a possible-malware download;
* execution of a process with the same filename;
* an outbound connection to the same destination;
* a Zeek reverse-shell start;
* reverse-shell movement;
* reverse-shell closure.

### Supported orders

Rules `120920-120923` cover the first sequence.

Rules `120924-120926` cover the alternative order in which the reverse-shell start is received before the complete malware and endpoint correlation.

### Severity progression

| Stage                                 | Rules                | Level |
| ------------------------------------- | -------------------- | ----: |
| Same-IP and same-filename correlation | `120920`             |     5 |
| Reverse-shell start                   | `120921` or `120924` |     8 |
| Reverse-shell movement                | `120922` or `120925` |    10 |
| Complete lifecycle                    | `120923` or `120926` |    12 |

## Correlation timeframes

|      Timeframe | Usage                                                            |
| -------------: | ---------------------------------------------------------------- |
|  `900` seconds | Initial malware, connection and reverse-shell-start correlations |
| `1800` seconds | Correlation with reverse-shell movement                          |
| `3600` seconds | Correlation with reverse-shell closure and full lifecycle        |

The windows account for:

* differences in event arrival time;
* delayed processing;
* long-lived reverse shells;
* time between payload download and execution;
* traffic occurring after initial access.

## Main correlation fields

| Field            | Source          | Purpose                                                                      |
| ---------------- | --------------- | ---------------------------------------------------------------------------- |
| `dest_ip`        | Zeek and Auditd | Associates the endpoint connection with the network destination              |
| `dest_port`      | Zeek and Auditd | Associates the process connection with the reverse-shell listener            |
| `file_name`      | Zeek and Auditd | Associates a downloaded file with the executed binary                        |
| `process_binary` | Auditd          | Provides the complete executable path                                        |
| `uid`            | Zeek            | Associates start, movement and final events from the same network connection |
| `src_ip`         | Zeek            | Identifies the victim or connection initiator                                |
| `orig_bytes`     | Zeek            | Indicates bytes sent by the originator                                       |
| `resp_bytes`     | Zeek            | Indicates bytes sent by the responder                                        |

## `1004_auditd_rules.xml`

This file contains custom rules for privileged-command execution and post-exploitation activity recorded through Auditd.

The rules belong to the following groups:

```text
local
auditd
```

They use the Wazuh parent rule:

```text
80792
```

and distinguish the events through:

```text
audit.key
```

The corresponding endpoint Auditd rules are stored in:

```text
blue-team/auditd/rules.d/privileged-command.rules
```

## Rule summary

| Rule ID  | Level | Audit key       | Purpose                                                    | MITRE ATT&CK                   |
| -------- | ----: | --------------- | ---------------------------------------------------------- | ------------------------------ |
| `100700` |    13 | `suid_exec`     | Detects a non-root user executing a root-effective command | `T1548.001`                    |
| `100701` |     5 | `root_exec`     | Detects commands executed from a direct root session       | —                              |
| `100702` |     7 | `sudo_exec`     | Detects execution of `sudo` by an authenticated user       | `T1548.003`                    |
| `100703` |     7 | `su_exec`       | Detects execution of `su` by an authenticated user         | `T1078`                        |
| `100704` |     5 | `download_exec` | Detects execution from the monitored Downloads directory   | `T1078` in the current ruleset |

## SUID execution

Rule `100700` detects events tagged with:

```text
suid_exec
```

The endpoint rule is designed to identify:

```text
uid != 0
euid = 0
```

This indicates that a process started by a non-root user executed with effective root privileges.

The event may be associated with:

* legitimate SUID programs;
* expected administrative utilities;
* SUID misconfiguration;
* SUID binary abuse;
* privilege-escalation attempts.

Because of its security impact, the rule uses level `13`.

## Root-session execution

Rule `100701` detects commands tagged with:

```text
root_exec
```

This key is generated for commands executed from a direct root session.

The event provides visibility into post-escalation activity, but it does not independently indicate malicious behavior.

Legitimate root administration can also trigger this rule.

## `sudo` execution

Rule `100702` detects events tagged with:

```text
sudo_exec
```

The alert includes:

* Audit user identifier;
* executable path.

The configured MITRE ATT&CK mapping is:

```text
T1548.003
```

## `su` execution

Rule `100703` detects events tagged with:

```text
su_exec
```

It records the use of `/usr/bin/su` by an authenticated user.

The event can support the reconstruction of:

* user switching;
* attempted privilege escalation;
* post-compromise account usage;
* legitimate administrative activity.

## Execution from Downloads

Rule `100704` detects execution of files located under:

```text
/home/client/Downloads
```

The event can be relevant when analyzing:

* downloaded payloads;
* malware execution;
* scripts obtained from external systems;
* reverse-shell payloads;
* temporary tools used after compromise.

Execution from the directory is not automatically malicious.

The event should be evaluated together with:

* filename;
* file hash;
* FIM events;
* previous Zeek download events;
* destination address;
* Auditd `rs_connect` events;
* reverse-shell correlations.

# Severity model

The custom rules use a progressive severity model.

| Level | General interpretation                              |
| ----: | --------------------------------------------------- |
|     0 | Suppressed, parent or informational event           |
|     3 | Low-severity operational information                |
|     5 | Initial suspicious event or endpoint evidence       |
|     7 | Relevant security event requiring investigation     |
|     8 | Correlated or higher-confidence suspicious behavior |
|    10 | Strong evidence of active malicious traffic         |
|    12 | Complete reverse-shell lifecycle                    |
|    13 | High-impact privilege-escalation evidence           |

Individual telemetry events are intentionally assigned lower levels than multi-source correlations.

This prevents a single network connection or command execution from being treated as conclusive evidence of compromise.

# Installation

Copy the custom rules to the Wazuh Manager:

```bash
sudo cp 001_zeek_rules.xml /var/ossec/etc/rules/
sudo cp 002_zeek_rules_custom.xml /var/ossec/etc/rules/
sudo cp 003_auditd_rev_shell.xml /var/ossec/etc/rules/
```

```bash
sudo chown wazuh:wazuh /var/ossec/etc/rules/001_zeek_rules.xml
```

# Validation

## Test individual events

Use:

```bash
sudo /var/ossec/bin/wazuh-logtest
```

Paste a real or sanitized event and verify:

```text
Phase 1: Completed pre-decoding
Phase 2: Completed decoding
Phase 3: Completed filtering (rules)
```

Check that the output contains:

* the expected decoder;
* the expected rule ID;
* the correct alert level;
* the required structured fields;
* the expected rule groups.

## Test correlation rules

Correlation rules cannot be fully validated using only one event.

A complete sequence must be reproduced, for example:

```text
100909 -> 110900 -> 100910 -> 100912 -> 100911
```

or:

```text
110900 -> 100910 -> 100912 -> 100911
```

The order used depends on the correlation chain being tested.

## Restart the manager

After adding or modifying the files:

```bash
sudo systemctl restart wazuh-manager
```

Verify its status:

```bash
sudo systemctl status wazuh-manager --no-pager
```

Check the manager log:

```bash
sudo grep -iE "error|warning|rules|decoder|invalid" \
  /var/ossec/logs/ossec.log | tail -n 100
```

# Troubleshooting

## Rule does not trigger

Check:

1. whether the expected decoder was selected;
2. whether the required fields were extracted;
3. whether the event contains the correct Auditd key;
4. whether the parent rule matched;
5. whether the custom file has the `.xml` extension;
6. whether the manager was restarted;
7. whether the rule timeframe expired;
8. whether events arrived from different agents without `global_frequency`;
9. whether `same_field` values are represented identically;
10. whether the source log path matches the `<location>` condition.

## Auditd connection is not detected

Verify:

```bash
sudo ausearch -k rs_connect -i
```

Confirm that the event contains:

* `success=yes`;
* the expected syscall number;
* a valid destination address;
* a non-zero destination port;
* the `rs_connect` key.

## Privileged event is not detected

Verify the active Auditd rules:

```bash
sudo auditctl -l
```

Search by key:

```bash
sudo ausearch -k suid_exec -i
sudo ausearch -k root_exec -i
sudo ausearch -k sudo_exec -i
sudo ausearch -k su_exec -i
sudo ausearch -k download_exec -i
```

Confirm that Wazuh extracts the key as:

```text
audit.key
```

## Correlation stops before the final rule

Check that:

* the initial correlated rule was generated;
* subsequent events arrived within the configured timeframe;
* Zeek events use the same connection UID;
* destination IP and port fields match;
* the events are visible in Wazuh alerts;
* all intermediate rules have actually triggered.

# Security and repository hygiene

The rule files do not contain authentication secrets.

The following files must not be committed:

* files ending in `.bak`;
* temporary test rules;
* Wazuh default examples not used by the laboratory;
* unredacted alert archives;
* credentials;
* passwords;
* enrollment secrets;
* authentication databases;
* private SSH keys;
* private TLS keys;
* API tokens;
* malicious payloads.

Private laboratory addresses and documented internal paths can remain when they are required to reproduce the environment.

The repository should contain only:

* active custom rules;
* reproducible configuration;
* sanitized evidence;
* documentation of validated scenarios.
