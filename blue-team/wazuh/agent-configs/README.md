# Wazuh Agent Configurations

This directory contains sanitized Wazuh Agent configurations used by the monitored hosts in the laboratory.

Only configurations containing relevant customizations are included. Default or unchanged agent configurations are not duplicated unless they are necessary to reproduce a scenario.

## Directory structure

```text
agent-configs/
├── README.md
├── client-linux-agent-ossec.conf
└── zeek-agent-ossec.conf
```

## Available configurations

| File                            | Host       | Purpose                                                                    |
| ------------------------------- | ---------- | -------------------------------------------------------------------------- |
| `zeek-agent-ossec.conf`         | `ZeekVM`   | Collects standard and custom Zeek logs                                     |
| `client-linux-agent-ossec.conf` | `ClientVM` | Collects endpoint logs, Auditd events and File Integrity Monitoring events |

## Client Linux Agent

The file:

```text
client-linux-agent-ossec.conf
```

contains the Wazuh Agent configuration deployed on `ClientVM`.

The original configuration is stored on the endpoint at:

```text
/var/ossec/etc/ossec.conf
```

### Manager connection

The agent sends events to the Wazuh Manager using:

```text
Address: 10.3.10.3
Port: 1514/TCP
Agent name: Client-Linux
```

The address belongs to the private laboratory network.

### Auditd log collection

The agent collects the Linux Audit log through the following configuration:

```xml
<localfile>
  <log_format>audit</log_format>
  <location>/var/log/audit/audit.log</location>
</localfile>
```

This allows Auditd events generated by the rules stored under:

```text
blue-team/auditd/rules.d/
```

to be forwarded to the Wazuh Manager.

Auditd provides host-based information such as:

* executed process;
* executable path;
* process identifier;
* parent process identifier;
* user and Audit user identifiers;
* monitored system call;
* socket address;
* Audit rule key.

These events complement the network-based evidence generated by Zeek during the reverse-shell scenario.

### File Integrity Monitoring

The following directory is monitored by the Wazuh File Integrity Monitoring module:

```text
/home/client/Downloads
```

The configuration uses:

```xml
<directories check_all="yes" whodata="yes">/home/client/Downloads</directories>
```

This enables the detection of file creation, modification and deletion events.

The `whodata` option provides additional information about the user and process responsible for the filesystem activity.

This monitoring is useful during scenarios involving:

* malware downloads;
* payload execution;
* file modification;
* file deletion;
* post-exploitation activity;
* data staging.

### Additional log sources

The agent also collects:

| Source                    | Format     |
| ------------------------- | ---------- |
| Systemd journal           | `journald` |
| Wazuh Active Response log | `syslog`   |
| Debian package log        | `syslog`   |
| Linux Audit log           | `audit`    |

The relevant paths are:

```text
journald
/var/ossec/logs/active-responses.log
/var/log/dpkg.log
/var/log/audit/audit.log
```

### Security monitoring modules

The configuration also enables standard Wazuh capabilities including:

* Rootcheck;
* System Inventory;
* Security Configuration Assessment;
* File Integrity Monitoring;
* Active Response;
* local command monitoring.

## Event flow

Auditd events follow this path:

```text
Audit rules
     |
     v
Linux Audit subsystem
     |
     v
/var/log/audit/audit.log
     |
     v
Wazuh Agent on ClientVM
     |
     v
Wazuh Manager
     |
     v
Decoders and detection rules
     |
     v
Wazuh alerts
```

File Integrity Monitoring events follow this path:

```text
File activity on ClientVM
     |
     v
Wazuh FIM
     |
     v
Wazuh Agent
     |
     v
Wazuh Manager
     |
     v
FIM alert
```

## Deployment

The configuration file must be placed on the endpoint at:

```text
/var/ossec/etc/ossec.conf
```

After modifying the configuration, restart the Wazuh Agent:

```bash
sudo systemctl restart wazuh-agent
```

Verify its status:

```bash
sudo systemctl status wazuh-agent --no-pager
```

Check the agent log for errors:

```bash
sudo tail -n 100 /var/ossec/logs/ossec.log
```

## Security and sanitization

The configuration stored in this repository does not include authentication secrets.

The following files must never be committed:

```text
/var/ossec/etc/client.keys
/var/ossec/etc/authd.pass
/var/ossec/etc/sslmanager.key
```

References to these files inside `ossec.conf` can remain because they only indicate local filesystem paths.

The following information must also be excluded from the repository:

* passwords;
* API tokens;
* private SSH keys;
* private TLS keys;
* enrollment secrets;
* authentication databases;
* unredacted personal information;
* production credentials.

Private laboratory IP addresses, agent names and documented local paths are retained to make the environment reproducible.
