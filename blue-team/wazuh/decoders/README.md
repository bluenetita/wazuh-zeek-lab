# Wazuh Decoders

This directory contains the custom XML decoders currently deployed on the Wazuh Manager.

The decoders transform raw events collected from the monitored endpoints into structured fields that can be used by Wazuh detection rules.

## Directory structure

```text
decoders/
├── README.md
├── 000_audit_saddr_decoder.xml
└── 001_apparmor_decoder.xml
```

## Available decoders

| File                          | Source                  | Purpose                                                                              |
| ----------------------------- | ----------------------- | ------------------------------------------------------------------------------------ |
| `000_audit_saddr_decoder.xml` | Auditd                  | Extracts process, user, destination IP and destination port from `rs_connect` events |
| `001_apparmor_decoder.xml`    | AppArmor through Auditd | Extracts structured fields from AppArmor audit events                                |

## Auditd reverse-shell decoder

The file:

```text
000_audit_saddr_decoder.xml
```

contains the decoder:

```text
auditd_rs_connect_enriched
```

The decoder is applied to Auditd events containing:

```text
key="rs_connect"
```

This key is generated by the Auditd rules documented in:

```text
blue-team/auditd/rules.d/reverse_shell.rules
```

The decoder processes successful `connect()` system-call events generated on `ClientVM`.

### Extracted fields

| Field            | Description                                     |
| ---------------- | ----------------------------------------------- |
| `audit_id`       | Linux Audit event identifier                    |
| `syscall`        | System-call number                              |
| `success`        | Result of the system call                       |
| `ppid`           | Parent process identifier                       |
| `pid`            | Process identifier                              |
| `auid`           | Audit user identifier                           |
| `uid`            | Process user identifier                         |
| `euid`           | Effective user identifier                       |
| `tty`            | Terminal associated with the event              |
| `ses`            | Audit session identifier                        |
| `process_name`   | Process name extracted from `comm`              |
| `process_binary` | Complete executable path                        |
| `file_name`      | Executable filename                             |
| `dest_ip`        | Destination IPv4 address extracted from `SADDR` |
| `dest_port`      | Destination port extracted from `SADDR`         |

### Expected event format

The decoder expects enriched Auditd events containing a structure similar to:

```text
SADDR={
  saddr_fam=inet
  laddr=DESTINATION_IP
  lport=DESTINATION_PORT
}
```

### Event flow

```text
Process on ClientVM
        |
        v
connect() system call
        |
        v
Linux Audit subsystem
        |
        v
/var/log/audit/audit.log
        |
        v
Wazuh Agent
        |
        v
Wazuh Manager
        |
        v
auditd_rs_connect_enriched
        |
        v
Wazuh detection rules
```

A successful `connect()` event does not by itself prove that a reverse shell is present.

The decoded information must be evaluated together with:

* the responsible executable;
* the parent process;
* destination address and port;
* event timestamp;
* related Zeek network evidence;
* other Wazuh alerts generated during the scenario.

## AppArmor decoder

The file:

```text
001_apparmor_decoder.xml
```

contains the `apparmor_audit` decoder family.

The decoder identifies events generated by AppArmor and transported through the Linux Audit subsystem.

It supports records containing types such as:

```text
AVC
USER_AVC
1400
1107
```

The decoder is autonomous and does not replace the standard Wazuh Auditd decoder.

### General fields

| Field                   | Description                                              |
| ----------------------- | -------------------------------------------------------- |
| `apparmor.audit_type`   | Audit record type                                        |
| `apparmor.audit_id`     | Audit event identifier                                   |
| `apparmor.event`        | AppArmor result, such as `ALLOWED`, `DENIED` or `STATUS` |
| `apparmor.operation`    | Operation requested by the process                       |
| `apparmor.object_class` | AppArmor object class                                    |
| `apparmor.profile`      | AppArmor profile involved                                |
| `apparmor.name`         | Resource involved in the operation                       |
| `apparmor.name2`        | Optional second resource                                 |
| `apparmor.pid`          | Process identifier                                       |
| `apparmor.command`      | Process command name                                     |

### Permission fields

| Field                | Description                    |
| -------------------- | ------------------------------ |
| `apparmor.requested` | Requested permissions          |
| `apparmor.denied`    | Denied permissions             |
| `apparmor.fsuid`     | Filesystem user identifier     |
| `apparmor.ouid`      | Resource owner user identifier |

### Diagnostic fields

| Field            | Description            |
| ---------------- | ---------------------- |
| `apparmor.info`  | Diagnostic information |
| `apparmor.error` | Error code             |

### Capability fields

| Field                      | Description       |
| -------------------------- | ----------------- |
| `apparmor.capability`      | Capability number |
| `apparmor.capability_name` | Capability name   |

### Signal and ptrace fields

| Field             | Description             |
| ----------------- | ----------------------- |
| `apparmor.signal` | Signal involved         |
| `apparmor.peer`   | Peer profile or process |

### Network fields

| Field                     | Description             |
| ------------------------- | ----------------------- |
| `apparmor.network_family` | Network address family  |
| `apparmor.socket_type`    | Socket type             |
| `apparmor.protocol`       | Network protocol number |

### D-Bus fields

### Mount and filesystem fields

## Installation

The decoder files must be placed on the Wazuh Manager under:

```text
/var/ossec/etc/decoders/
```

Example:

```bash
sudo cp 000_audit_saddr_decoder.xml /var/ossec/etc/decoders/
sudo cp 001_apparmor_decoder.xml /var/ossec/etc/decoders/
```

Set the expected ownership and permissions:

```bash
sudo chown wazuh:wazuh /var/ossec/etc/decoders/000_audit_saddr_decoder.xml
sudo chown wazuh:wazuh /var/ossec/etc/decoders/001_apparmor_decoder.xml

sudo chmod 660 /var/ossec/etc/decoders/000_audit_saddr_decoder.xml
sudo chmod 660 /var/ossec/etc/decoders/001_apparmor_decoder.xml
```

## Testing

Custom decoders should be tested using:

```bash
sudo /var/ossec/bin/wazuh-logtest
```

Paste a real or sanitized event and verify the decoding phase:

```text
Phase 2: Completed decoding
```

The output should contain:

* the expected decoder name;
* the required extracted fields;
* values consistent with the original event.

## Applying changes

After adding or modifying a decoder:

```bash
sudo systemctl restart wazuh-manager
```

Verify the manager status:

```bash
sudo systemctl status wazuh-manager --no-pager
```

Check for configuration errors:

```bash
sudo grep -iE "error|decoder|invalid" /var/ossec/logs/ossec.log | tail -n 100
```

## Relationship with Wazuh rules

Decoders do not directly generate alerts.

The complete processing flow is:

```text
Raw event
    |
    v
Wazuh decoder
    |
    v
Structured fields
    |
    v
Wazuh rule
    |
    v
Alert
```

The rules associated with these decoders are stored under:

```text
blue-team/wazuh/rules/
```

Changing a field name in a decoder may require updating every Wazuh rule that references that field.

## Excluded files

The following files must not be committed:

* `.bak` backup files;
* disabled decoder copies;
* temporary decoder versions;
* complete unredacted logs;
* diagnostic output;
* credentials;
* tokens;
* private keys;
* personal or confidential information.

This directory contains only the active custom decoders required by the current laboratory configuration.
